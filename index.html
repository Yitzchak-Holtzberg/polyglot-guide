Go strips away most of what C# layers on top of a program. No namespaces on every file, no classes required, no semicolons. The mental model shift is: Go is closer to C with a GC, modern concurrency primitives, and strong opinions about simplicity.

Program entry point
Hello World
C#
// Top-level statements (C# 9+)
Console.WriteLine("Hello, World!");

// Or the classic form:
namespace MyApp;
class Program {
    static void Main(string[] args) {
        Console.WriteLine("Hello, World!");
    }
}
Go
package main

import "fmt"

func main() {
    fmt.Println("Hello, World!")
}
Go idiomEvery executable must be in package main with a main() func. No class wrapper, no namespace. Packages are directories — the package name is declared at the top of every file in the folder.

Go idiomGo has only one loop keyword: for. No while, no foreach, no do-while. The range keyword iterates slices, maps, channels, and strings. Switch cases don't fall through automatically — the opposite of C.
Biggest mental shiftNo semicolons. No parentheses on if/for/switch. Opening braces must be on the same line (Go's parser inserts semicolons at line endings — a different-line brace causes a syntax error). gofmt enforces all of this for you.

Pointers
PointersGo has them. C# mostly hides them.
C#
// C# classes are reference types — "pointers" implicitly
class Point { public int X, Y; }

Point p = new Point { X = 1, Y = 2 };
Point q = p;   // q and p point at same object
q.X = 99;      // p.X is also 99

// ref keyword for struct pass-by-ref
void Increment(ref int n) { n++; }
Go
// Everything is a value by default — use & to get a pointer
type Point struct { X, Y int }

p := Point{X: 1, Y: 2}
q := p         // q is a copy
q.X = 99       // p.X is still 1

ptr := &p      // *Point — address of p
ptr.X = 99    // p.X is now 99 (auto-deref)

func increment(n *int) { *n++ }  // explicit pointer
Critical differenceIn C#, class instances pass by reference automatically; structs pass by value. In Go, everything passes by value. To mutate a caller's struct, you pass a *Point (pointer). This is explicit and intentional — no hidden reference semantics.
Zero values
Zero Values (Go's answer to null)
C#

    public string? Name;    // null
    public int Age;          // 0
    public bool Active;      // false
}

// NullReferenceException is a real concern

Go
// Every type has a zero value — always safe to use
type User struct {
    Name   string   // ""
    Age    int      // 0
    Active bool     // false
}

var u User

Go idiomZero values are a feature. var mu sync.Mutex is immediately usable. var b bytes.Buffer is ready to write to. Design your types so the zero value is a valid starting state — this is idiomatic Go.

Go functions are first-class and can return multiple values. This replaces most uses of out/ref parameters and is the foundation of Go's error handling pattern.

Basic Functions
C#
static int Add(int a, int b) => a + b;

static string Greet(string name = "World") {
    return $"Hello, {name}!";
}

// Multiple returns via tuple
static (int Min, int Max) MinMax(int[] arr) {
    return (arr.Min(), arr.Max());
}

// out parameter
static bool TryParse(string s, out int result) {
    return int.TryParse(s, out result);
}
Go
func add(a, b int) int { return a + b }

func greet(name string) string {
    return "Hello, " + name + "!"
}

// Multiple returns — native, no tuple wrapper
func minMax(arr []int) (int, int) {
    mn, mx := arr[0], arr[0]
    for _, v := range arr {
        if v < mn { mn = v }
        if v > mx { mx = v }
    }
    return mn, mx
}

// "TryParse" pattern: (value, error)
func tryParse(s string) (int, error) {
    return strconv.Atoi(s)
}
Go idiomNo default parameters in Go. Use functional options pattern or separate functions for variants. No method overloading either — the language prefers explicit naming over implicit dispatch.
First-Class Functions & Closures
C#
// Func/Action delegates

// Closure

        evens = append(evens, n*2)

}
No LINQ in GoGo has no built-in map/filter/reduce. Go 1.23+ adds slices and maps stdlib packages with some helpers, but the idiomatic answer is usually a plain for range loop. It's more verbose but more readable to Go developers.
defer (Go-only)No C# equivalent
C#
// using statement for cleanup
using var file = File.Open("data.txt", FileMode.Open);
// file.Dispose() called at end of scope

// try/finally for guaranteed execution
try {
    mu.WaitOne();
    // ... work ...
} finally {
    mu.ReleaseMutex();
}
Go
// defer runs at function return — LIFO order
file, _ := os.Open("data.txt")
defer file.Close()   // runs when function returns

mu.Lock()
defer mu.Unlock()    // always runs, even on panic

// Multiple defers stack (LIFO)
defer fmt.Println("third")
defer fmt.Println("second")
defer fmt.Println("first")  // prints: first, second, third
Go idiomdefer is the Go replacement for using, finally, and RAII. It's attached to the function, not a block. The pattern Lock(); defer Unlock() appears everywhere in Go code.

Go has no classes. It has structs with methods. The mental model: take everything you put in a C# class and split it — data goes in the struct, behavior goes in functions attached to that struct via a receiver.

Structs & Methods
C#
class Rectangle {
    public double Width { get; set; }
    public double Height { get; set; }

    public Rectangle(double w, double h) {
        Width = w; Height = h;
    }

    public double Area() => Width * Height;

    public override string ToString() =>
        $"Rectangle({Width}x{Height})";
}

var r = new Rectangle(3, 4);
Console.WriteLine(r.Area());    // 12
Go
type Rectangle struct {
    Width  float64
    Height float64
}

// No constructor — use a factory func by convention
func NewRectangle(w, h float64) Rectangle {
    return Rectangle{Width: w, Height: h}
}

// Method: receiver goes before the func name
func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

func (r Rectangle) String() string {
    return fmt.Sprintf("Rectangle(%.0fx%.0f)", r.Width, r.Height)
}

r := NewRectangle(3, 4)
fmt.Println(r.Area())           // 12
Value vs Pointer ReceiversReplaces ref/mutable in C#
C#
// Class: mutation always works (reference type)
class Counter {
    public int Value;
    public void Increment() => Value++;
}

// Struct: must use ref to mutate
struct Point {
    public int X;
    void MoveRight() => X++;  // mutates a copy in C#!
}
Go
type Counter struct { Value int }

// Value receiver — gets a copy, can't mutate caller
func (c Counter) Get() int { return c.Value }

// Pointer receiver — mutates the original
func (c *Counter) Increment() { c.Value++ }

c := Counter{}
c.Increment()     // Go auto-takes address: (&c).Increment()
fmt.Println(c.Get()) // 1

// Rule of thumb: if any method mutates, all methods
// should use pointer receivers for consistency
Embedding (Composition over Inheritance)
C#
class Animal {
    public string Name { get; set; }
    public void Breathe() => Console.WriteLine("Breathing");
}

// Inheritance
class Dog : Animal {
    public void Bark() => Console.WriteLine("Woof");
}

var d = new Dog { Name = "Rex" };
d.Breathe();  // inherited
d.Bark();
Go
type Animal struct { Name string }
func (a Animal) Breathe() { fmt.Println("Breathing") }

// Embedding — NOT inheritance, but promotes fields/methods
type Dog struct {
    Animal            // embedded (no field name)
    Breed string
}

func (d Dog) Bark() { fmt.Println("Woof") }

d := Dog{Animal: Animal{Name: "Rex"}, Breed: "Lab"}
d.Breathe()   // promoted — works directly
d.Name        // also promoted
d.Bark()
No inheritance in GoGo has no extends, no virtual methods, no override, no base class. Embedding promotes methods but doesn't create an IS-A relationship. This forces composition patterns that most C# devs eventually prefer anyway.

Go interfaces are implicit — a type satisfies an interface simply by having the right methods. You never write implements. This is called structural typing and it completely changes how you design abstractions.

Interface Definition & Satisfaction
C#
// Explicit declaration required
interface IWriter {
    void Write(string data);
}

// Explicit implementation
class FileWriter : IWriter {
    public void Write(string data) {
        File.AppendAllText("log.txt", data);
    }
}

IWriter w = new FileWriter();
w.Write("hello");
Go
// Interface defined anywhere — no coupling to implementors
type Writer interface {
    Write(data string)
}

// FileWriter satisfies Writer implicitly — no "implements"
type FileWriter struct{ Path string }

func (f FileWriter) Write(data string) {
    os.WriteFile(f.Path, []byte(data), 0644)
}

var w Writer = FileWriter{Path: "log.txt"}
w.Write("hello")
Go idiomDefine interfaces at the point of use, not the point of implementation. In C# you define IWriter in the library and consumers depend on it. In Go, consumers define the interface they need — libraries just export concrete types. This inverts the dependency direction and makes mocking trivial.

This is the biggest culture shock coming from C#. Go has no exceptions. Errors are values — functions return them, callers check them. It feels verbose at first; experienced Go devs consider it a feature.

Error Handling
C#
// Exceptions bubble up automatically
try {
    var data = File.ReadAllText("data.txt");
    var n = int.Parse(data.Trim());
    Console.WriteLine(n * 2);
}
catch (FileNotFoundException ex) {
    Console.WriteLine($"File not found: {ex.Message}");
}
catch (FormatException) {
    Console.WriteLine("Bad number format");
}
Go
// Errors are return values — check immediately
data, err := os.ReadFile("data.txt")
if err != nil {
    fmt.Println("File not found:", err)
    return
}

n, err := strconv.Atoi(strings.TrimSpace(string(data)))
if err != nil {
    fmt.Println("Bad number format")
    return
}

fmt.Println(n * 2)
No exceptionsThere is no throw, no catch, no try. There is panic() (like an unhandled exception) and recover() (like a last-resort catch in a deferred func), but these are reserved for truly unrecoverable situations — not normal control flow.
Custom Errors
C#
class ValidationException : Exception {
    public string Field { get; }
    public ValidationException(string field, string msg)
        : base(msg) { Field = field; }
}

throw new ValidationException("email", "invalid");

// Caller
catch (ValidationException ex) when (ex.Field == "email") {
    // ...
}
Go
type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("%s: %s", e.Field, e.Message)
}

// Return, not throw
return &ValidationError{Field: "email", Message: "invalid"}

// Caller uses errors.As for type inspection
var ve *ValidationError
if errors.As(err, &ve) && ve.Field == "email" {
    // ...
}
Go idiomAny type that implements Error() string satisfies the built-in error interface. Use errors.Is() for sentinel errors, errors.As() for typed errors — mirrors C#'s is/catch patterns but explicit.

Go's concurrency model is its killer feature. Goroutines are like threads but cost ~2KB each (vs ~1MB for OS threads). Channels are typed, synchronized communication pipes. The slogan: "Don't communicate by sharing memory; share memory by communicating."

Goroutines vs Tasks
C#
// async/await
async Task DoWorkAsync() {
    await Task.Delay(1000);
    Console.WriteLine("done");
}

// Fire and forget (bad practice, but exists)
_ = DoWorkAsync();

// Wait for multiple tasks
await Task.WhenAll(
    DoWorkAsync(),
    DoWorkAsync()
);
Go
// Goroutine — just add "go"
func doWork() {
    time.Sleep(time.Second)
    fmt.Println("done")
}

go doWork()   // fire and forget

// Wait for multiple goroutines — WaitGroup
var wg sync.WaitGroup
for i := 0; i < 2; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        doWork()
    }()
}
wg.Wait()
No async/awaitGo has no async keyword. All I/O is blocking at the Go level — the runtime transparently suspends goroutines waiting on I/O and runs other goroutines. You write synchronous-looking code and get async behavior for free.
ChannelsNo direct C# equivalent
C#
// Closest analog: Channel<T> from System.Threading.Channels
var ch = Channel.CreateUnbounded<int>();

// Producer
_ = Task.Run(async () => {
    await ch.Writer.WriteAsync(42);
    ch.Writer.Complete();
});

// Consumer
await foreach (var item in ch.Reader.ReadAllAsync()) {
    Console.WriteLine(item);
}
Go
// Channels are a first-class language feature
ch := make(chan int)           // unbuffered
buf := make(chan int, 10)       // buffered (cap 10)

// Producer goroutine
go func() {
    ch <- 42   // send
    close(ch)  // signal done
}()

// Consumer
for v := range ch {   // receives until closed
    fmt.Println(v)
}

// select — like switch but for channels
select {
case v := <-ch1:   fmt.Println("ch1:", v)
case ch2 <- 99:   fmt.Println("sent to ch2")
case <-time.After(1 * time.Second):
    fmt.Println("timeout")
}
selectis Go's multiplexer for channels — it picks whichever case is ready, blocking until one is. This is how you implement timeouts, cancellation, and fan-in patterns without any framework.

Go's primary collection types are slices (dynamic arrays) and maps. There's no List<T>, Dictionary, LinkedList, etc. from a rich BCL — you use slices for almost everything.

Arrays & SlicesSlice ≈ List<T>
C#
// Array — fixed size
int[] arr = { 1, 2, 3 };

// List — dynamic
var list = new List<int> { 1, 2, 3 };
list.Add(4);
list.RemoveAt(0);

// Slice/span
var span = arr[1..3];    // [2, 3]

// Length
int len = list.Count;
Go
// Array — fixed size, rarely used directly
arr := [3]int{1, 2, 3}

// Slice — use this for everything
s := []int{1, 2, 3}
s = append(s, 4)          // add to end
s = append(s[1:], s[2:]...) // delete index 0 (tricky!)

// Sub-slice (shares underlying array!)
sub := s[1:3]              // [2, 3] — not a copy

// Length and capacity
len(s)
cap(s)

// Pre-allocate
s2 := make([]int, 0, 100)  // len=0, cap=100
Slice gotchaSub-slices share the underlying array. Mutating sub[0] mutates s[1]. Use copy() when you need independence. append may or may not allocate a new array depending on capacity — this is the same as List<T> doubling behavior internally.

Go idiomThe comma-ok pattern (val, ok := m[key]) appears everywhere in Go — map lookups, type assertions, channel receives. It's the idiomatic way to distinguish "zero value" from "not present."

Go added generics in 1.18 (2022). They're more constrained than C# generics — no variance, no generic methods on types, simpler — but cover the important use cases.

Generic Functions & Types
C#
// Generic function
static T Max<T>(T a, T b) where T : IComparable<T>
    => a.CompareTo(b) > 0 ? a : b;

// Generic class
class Stack<T> {
    private List<T> _items = new();
    public void Push(T item) => _items.Add(item);
    public T Pop() {
        var top = _items[^1];
        _items.RemoveAt(_items.Count - 1);
        return top;
    }
}
Go
// Constraint interface
type Ordered interface {
    ~int | ~float64 | ~string   // union of types
}

// Generic function
func Max[T Ordered](a, b T) T {
    if a > b { return a }
    return b
}

// Generic struct
type Stack[T any] struct {
    items []T
}

func (s *Stack[T]) Push(item T) {
    s.items = append(s.items, item)
}

func (s *Stack[T]) Pop() T {
    n := len(s.items)
    top := s.items[n-1]
    s.items = s.items[:n-1]
    return top
}
Constraint syntaxGo constraints use interface syntax. ~int means "int or any type whose underlying type is int" — this lets user-defined types like type MyInt int satisfy the constraint. The cmp and slices packages in stdlib use this extensively.
What Go generics can't do (yet)No generic methods on non-generic types (you can't add a type parameter to a single method on a struct). No covariance/contravariance. No runtime type constraints beyond what the interface specifies. For most practical uses — generic containers, utility functions — they work well.

Ada was designed for safety-critical systems — aerospace, defense, medical devices. Where C# optimizes for developer productivity, Ada optimizes for correctness and explicitness. Everything that could be ambiguous is made explicit. Case-insensitive keywords, verbose syntax, and mandatory structure are features, not bugs.

Program structure
Hello World
C#
// Top-level statements (C# 9+)
Console.WriteLine("Hello, World!");

class Program {
    static void Main(string[] args) {
        Console.WriteLine("Hello, World!");
    }
}
Ada

begin
   Ada.Text_IO.Put_Line("Hello, World!");
end Hello;  -- end must repeat the name
Ada idiomEvery begin/end block names itself at the closing end. This is mandatory for procedures, functions, packages, and tasks — it makes large files scannable without an IDE. Ada is case-insensitive; BEGIN = begin = Begin.

Ada idiomAda's case statement is exhaustive — you must cover every possible value of the type (or use when others). This is checked at compile time. Missing a case is a compile error, not a runtime bug.

Ada distinguishes between procedures (no return value) and functions (returns a value). Both are called "subprograms." Parameters have explicit modes: in (read-only), out (write-only), and in out (read-write). This replaces C#'s ref/out and makes intent explicit in the signature.

Procedures & Functions

begin
   return A + B;

Ada has exceptions, but they're used much less than in C# because the type system catches so many bugs at compile time. Exceptions in Ada are declarations — they're not classes, they don't carry data by default, and they can't be subclassed.

Ada's concurrency model is built into the language itself — no external library. Tasks are language-level constructs (like goroutines but with more structure). Protected objects replace mutexes. The model is based on Communicating Sequential Processes theory, the same foundation as Go's channels.

C#

    Console.WriteLine("Running in task");
});

   entry Start;   -- rendezvous entry point
end Worker;

task body Worker is
begin
   accept Start;  -- wait for caller to rendezvous
   Ada.Text_IO.Put_Line ("Running");

RendezvousAda tasks communicate via rendezvous — a synchronous handshake where caller and task meet at an entry/accept point. Both block until the other arrives, data is exchanged, then both continue. This is safer than shared memory because there's no window where data is partially written.
Protected ObjectsBuilt-in mutex + monitor

    private readonly object _lock = new();
    private int _count = 0;

    }

    }
}
Ada
-- Protected object: language-level monitor
protected Safe_Counter is
   procedure Increment;         -- exclusive write access
   function  Value return Integer; -- concurrent read access
   entry     Wait_Until_Ten;    -- blocks until condition
private
   Count : Integer := 0;
end Safe_Counter;

protected body Safe_Counter is
   procedure Increment is
   begin  Count := Count + 1;  end;

   function Value return Integer is
   begin  return Count;  end;

   entry Wait_Until_Ten
      when Count >= 10 is     -- barrier condition
   begin  null;  end;
end Safe_Counter;
Protected entries have barriersAn entry in a protected object has a when guard (barrier condition). Callers block until the barrier is true — this is a language-level condition variable, replacing Monitor.Wait/Pulse. Procedures get exclusive lock. Functions get shared read lock. All enforced by the runtime, not convention.

Ada 2012 added contract-based programming via Pre, Post, and Type_Invariant aspects. These are similar to Code Contracts in C# but built into the language, checked at compile time where possible, and enforced at runtime otherwise. This is the feature that makes Ada so attractive for formal verification.

Ada idiomContracts live in the spec (the public interface), not the body (the implementation). Callers can see the contract without seeing the code. Tools like SPARK (a formal subset of Ada) can prove contracts correct statically — no runtime overhead, no test needed, mathematically guaranteed.

You add contracts and the tool proves, at compile time, that your code can never violate them — no runtime checks at all. Used in Eurofighter Typhoon avionics, NVIDIA GPU drivers, and cryptographic libraries. C# has no equivalent.

Ada's module system is called packages. A package has a spec (the public interface, like a .h file or interface definition) and a body (the implementation). The separation is enforced — callers can only see what the spec declares. This is the foundation of Ada's encapsulation model.

Package Spec & BodyLike C# namespace + class
C#

public class Circle {
    private readonly double _radius;

    public Circle(double radius) { _radius = radius; }

    public double Area()        => Math.PI * _radius * _radius;
    public double Perimeter()  => 2 * Math.PI * _radius;
}
Ada
-- geometry.ads (spec file — the interface)
package Geometry is
   type Circle is private;   -- opaque to callers

   function  Make_Circle (Radius : Float) return Circle;
   function  Area       (C : Circle)      return Float;
   function  Perimeter  (C : Circle)      return Float;
private
   type Circle is record
      Radius : Float;
   end record;
end Geometry;

-- geometry.adb (body file — the implementation)
package body Geometry is
   Pi : constant := 3.14159265;

   function Make_Circle (Radius : Float) return Circle is
   begin  return (Radius => Radius);  end;

   function Area (C : Circle) return Float is
   begin  return Pi * C.Radius ** 2;  end;

   function Perimeter (C : Circle) return Float is
   begin  return 2.0 * Pi * C.Radius;  end;
end Geometry;
Spec/body separationAda enforces the interface/implementation split at the file level. Callers compile against the spec (.ads) and never see the body (.adb). Changing the body doesn't require recompiling callers. This is what C# interfaces enforce by convention — Ada enforces it physically.

Three-valued logicIn SQL, every boolean expression evaluates to TRUE, FALSE, or UNKNOWN. NULL = NULL evaluates to UNKNOWN, not TRUE. A WHERE clause only passes rows where the condition is TRUE — UNKNOWN rows are filtered out silently. This is the most common source of subtle SQL bugs.

ON users (email);

-- Clustered index (determines physical row order; one per table)
CREATE CLUSTERED INDEX ix_orders_date
ON orders (created_at);

-- Covering index: INCLUDE columns avoid key lookups
CREATE NONCLUSTERED INDEX ix_orders_customer_covering
ON orders (customer_id, status)
INCLUDE (amount, created_at);  -- in leaf pages, not key

ON orders (created_at)
WHERE status = 'pending';     -- tiny, very fast for hot path
Covering indexesA covering index contains all columns the query needs, eliminating the "key lookup" (the extra read back to the clustered index). For a query SELECT amount, created_at FROM orders WHERE customer_id = 42 AND status = 'shipped', an index on (customer_id, status) INCLUDE (amount, created_at) answers the entire query from the index alone. This is often a 10x speedup.

// Nested Loops    — join with loops (small sets, indexed)
// Merge Join      — join on pre-sorted sets (fast)

Pointers are the core of C. Everything else is sugar. A pointer is a variable that holds a memory address. Once you understand that one sentence, all of C follows from it. C# has pointers too (in unsafe blocks) but hides them from you most of the time. In C, you live with them.

Memory management is where C diverges most sharply from C#. The GC doesn't exist. You allocate with malloc, you free with free, and you never forget. Every memory bug (use-after-free, double-free, leak, buffer overflow) that C# protects you from, C lets you commit. This is also why C is fast.

    int x = 42;
    return &x;   /* DANGLING POINTER — undefined behavior */
}

/* Heap — manual allocation */

The three rules of malloc1. Always check the return value — malloc returns NULL on OOM. 2. Always free exactly once — double-free is undefined behavior. 3. Never use after free — set pointer to NULL after freeing. Every C memory bug violates one of these three rules.

Undefined behavior (UB) is the most important concept in C for a C# developer to understand. UB means the C standard gives no guarantees about what happens — the compiler is free to assume UB never occurs, and may optimize in ways that seem impossible. UB doesn't crash — it silently corrupts, or disappears, or changes behavior based on optimization level.

Common undefined behaviors
C# — these throw exceptions
// All of these throw in C# — not UB

(Animal)obj;            // InvalidCastException
int.MaxValue + 1;       // checked: OverflowException
null.ToString();         // NullReferenceException
x / 0;                  // DivideByZeroException
C — these are undefined behavior
/* Out-of-bounds array access */
arr[100] = 5;           /* UB — may corrupt, may crash, may "work" */

/* Signed integer overflow */
int x = INT_MAX;
x++;                    /* UB — NOT guaranteed to wrap */
                        /* compiler may assume this never happens */

/* Use after free */
int *p = malloc(sizeof(int));
free(p);
*p = 42;               /* UB — silent, may crash later */

/* NULL dereference */
int *p = NULL;
*p = 5;                /* UB — usually crashes, not guaranteed */

/* Uninitialized variable */
int x;
printf("%d\n", x);     /* UB — any value, including "clean" looking ones */

/* Integer divide by zero */
int z = 10 / 0;        /* UB — no guaranteed exception */
UB is invisible and dangerousThe optimizer sees UB as a proof that a code path is unreachable. If you have if (x + 1 > x), the optimizer removes it (signed overflow is UB, so overflow never happens, so the condition is always true, so it's dead code). Programs with UB may appear to work with -O0 and silently break with -O2. Use -fsanitize=address,undefined in development.

Functions in OCaml are first-class values. Every function is automatically curried — a function of two arguments is actually a function of one argument that returns a function of one argument. This isn't a quirk, it's the foundation of partial application and higher-order programming.

Algebraic data types (ADTs) — specifically sum types (discriminated unions) — are OCaml's most powerful feature. They let you model data that can be one of several distinct shapes, with the compiler forcing you to handle every case. C# added discriminated unions partially via pattern matching on classes, but OCaml's are more expressive and checked exhaustively.

Pattern matching in OCaml is pervasive and deeply integrated. It's not just switch — it's the primary way to destructure any data. You match on variants, tuples, records, lists, integers, strings, and combinations. The compiler verifies exhaustiveness and reachability for every match expression.

Discriminated unions (DUs) are F#'s most powerful type construct. They model data that is one of several shapes — like a tagged union in C. Unlike C#'s abstract class hierarchies, DUs are exhaustively checked by the compiler in every match expression. Adding a new case shows you every place that needs updating.

Discriminated Unions

};

Computation expressions (CEs) are F#'s generalization of async/await, LINQ, and monadic programming. They let you define custom "builder" types that give new meaning to let!, return, yield, for, and other keywords inside a { } block. F#'s async { }, task { }, and seq { } are all computation expressions.

Custom Computation Expression
C# — no equivalent
// C# has async/await (one fixed CE)
// No way to create custom ones

// The closest: LINQ query syntax

        where x > 0
        select x * 2;
// But this only works for IEnumerable/IQueryable
F#
// Result CE — railway-oriented programming
type ResultBuilder() =
    member _.Return x = Ok x
    member _.Bind(x, f) =
        match x with
        | Ok v    -> f v
        | Error e -> Error e

let result = ResultBuilder()

// Now let! short-circuits on Error
let processOrder orderId = result {
    let! order    = getOrder    orderId    // Error short-circuits
    let! customer = getCustomer order.CustId
    let! _        = validateStock order
    return { order with Status = "confirmed" }
}
// If any step returns Error, the whole thing is Error
// No try/catch needed — errors flow through the type
Railway-oriented programmingThe result CE makes error handling declarative. Each let! step either succeeds and continues, or short-circuits with the error. This is the F# equivalent of Go's if err != nil { return err } pattern — but without the repetition. The entire computation is error-aware by construction.

Borrowing lets you use a value without taking ownership. A reference is a pointer the compiler guarantees is always valid. The borrow checker enforces two rules: you can have any number of shared (immutable) references, OR exactly one mutable reference — never both at the same time. This eliminates data races at compile time.

Lifetimes are the borrow checker's way of tracking how long references are valid. Most of the time Rust infers them and you never write them. You write explicit lifetime annotations when a function returns a reference and the compiler can't figure out which input it came from. Lifetimes don't change when memory is freed — they just let the compiler verify that references never outlive what they point to.

Lifetime Annotations
C#
// C# has no lifetime concept — GC handles it
// Dangling references are impossible (managed code)

    => a.Length >= b.Length ? a : b;

// No annotation needed — strings are GC'd objects
// They stay alive as long as anything references them
Rust
// Returning a reference — compiler needs to know
// which input the output reference came from

// 'a is a lifetime parameter — reads "lifetime a"
fn longer<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}
// 'a means: output lives as long as the shorter of a and b
// Caller is guaranteed the result is valid while both inputs are

// When lifetimes are obvious, compiler infers them (elision)
fn first_word(s: &str) -> &str {  // 'a inferred
    let bytes = s.as_bytes();
    for (i, &b) in bytes.iter().enumerate() {
        if b == b' ' { return &s[0..i]; }
    }
    &s[..]
}
Lifetime elision rulesThe compiler infers lifetimes in most cases via three elision rules. You only need explicit 'a annotations when: (1) returning a reference from a function with multiple reference inputs, (2) implementing methods that return references tied to struct fields, (3) using 'static for references that live for the entire program. Most Rust code has very few explicit lifetime annotations.

Traits are Rust's interface system. They define shared behaviour that types can implement. Unlike C# interfaces, traits can provide default implementations, can be implemented for types you don't own (coherence rules apply), and are resolved statically at compile time (monomorphization) unless you use dynamic dispatch (dyn Trait).

Trait definition & implementation Core Rust pattern

    fn sound(&self) -> &str;

    fn describe(&self) -> String {   // default impl

const buffer = try a.alloc(u8, 1024);
defer a.free(buffer);   // runs at scope exit
Pointers are typed and non-null*T points at one T, [*]T at many, *const T is read-only. There is no null pointer — absence is an optional (?*T). Take an address with &, dereference with .*.
No hidden allocationsAny function that needs memory takes an allocator argument, and you choose the strategy — page_allocator, ArenaAllocator, GeneralPurposeAllocator (which also catches leaks). The idiom: every alloc gets a defer free right beneath it.

Zig had non-nullable-by-default from day one — the thing C# bolted on with nullable reference types. A plain T can never be null; only ?T can, and the compiler forces you to handle the empty case before you touch the value.

Optionals vs nullable
C#
// Nullable reference / value types

// Null-coalescing & null-conditional

    Console.WriteLine(name.Length);
Zig

No exceptions. Errors are ordinary values drawn from an error set, and the only "magic" is try — sugar for "unwrap or return the error to my caller." There's no stack unwinding and no invisible throw three layers down.

        throw new FormatException("bad");

Zig
// An error set + an error union return

    if (s.len == 0) return ParseError.BadInt;
    return std.fmt.parseInt(i32, s, 10) catch
        return ParseError.BadInt;

Zig has structs, not classes — no inheritance, no hidden vtable, no this. A "method" is just a function whose first parameter is self, and calling p.dist() is plain sugar. The payoff shows up in the next topic: types are values.

Here's Zig's big idea. There's no separate generics system, no templates, no macros — just comptime: code the compiler runs at compile time. Because type is itself a value, a "generic type" is a plain function that takes a type and returns one.

Haskell's type inference is total — the compiler figures out the most general type of everything, so signatures are documentation rather than a requirement. The surprise for a C# dev is the =>: a typeclass constraint, not an argument.

Primitives & inference
C#
int    a   = 42;
long   big = 9_000_000_000;
double d   = 3.14;
bool   ok  = true;
char   c   = 'x';

Haskell

Inference is total (Hindley–Milner)Leave off a signature and GHC infers the most general type. inc becomes Num a => a -> a: "for any numeric type a, a function from a to a." Signatures then act as machine-checked documentation.
Int vs Integer, and that =>Int is a fast fixed-width word that can overflow; Integer is an unbounded bignum. The => is a constraint, not a parameter — read Num a => as "given that a is a number," which is the typeclass mechanism from two topics down.

Every function takes exactly one argument. A "two-argument" function is really a function that returns a function — which is why partial application and composition need no special syntax; they fall out of the type.

Curried by defaultadd :: Int -> Int -> Int is sugar for Int -> (Int -> Int) — a function returning a function. So add 5 is itself a value (a function from Int to Int), and partial application needs zero ceremony.

Algebraic data types are the heart of modeling in Haskell — and the feature C# spent years catching up to with records and switch expressions. data builds both "and" types (records) and "or" types (sums) in one keyword.

Typeclasses look like interfaces but aren't. The crucial difference: a type's membership in a class is declared separately from the type, so you can retrofit types you don't own into classes you don't own.

Typeclasses vs interfaces

    public string Show() => "$$$";
}

    => Console.WriteLine(x.Show());
Haskell

class Show' a where
  show' :: a -> String

-- The instance is SEPARATE from the data type
data Money = Money Int

  show' (Money n) = "$" ++ show n

-- Constraint: 'any a that is Show''-able'

Instances are decoupled from typesBecause instance Show' Money lives apart from data Money, you can add behavior to a type defined in another library. Eq, Ord, Show, Num are the everyday classes; this is closer to Rust traits or F# type extensions than to C# interfaces.
Resolved by type, at compile timeThere's no vtable riding on the object — show' (Money 5) selects the instance from the argument's type. And deriving (Eq, Ord, Show) on an ADT auto-generates the obvious instances, so you rarely write them by hand.

Haskell has no null and doesn't use exceptions for ordinary failure. Absence and error are values with types — Maybe and Either — so the compiler forces every caller to deal with the empty case.

Maybe & Either vs null/exceptions
C#

    id == 1 ? "ok" : null;

// ...exceptions for failure

    if (!int.TryParse(s, out var v))
        throw new FormatException();
    return v;
}
Haskell
-- Maybe a = Nothing | Just a
find :: Int -> Maybe String
find 1 = Just "ok"
find _ = Nothing

-- Either e a = Left err | Right value
parse :: String -> Either String Int
parse s = case reads s of
  [(n, "")] -> Right n
  _         -> Left "bad int"
Failure lives in the typeMaybe a is Nothing or Just x; Either e a is Left err or Right value (mnemonic: Right is the "right" answer). Since the result type advertises that it can fail, callers must pattern-match the value out before using it — no forgotten null check compiles.
_ is the wildcardfind _ = Nothing matches any other input; case … of does the same matching inline. reads is the safe parser, and the pattern [(n, "")] means "parsed an n with no leftover input." Later you'll chain these with do instead of nesting cases.

Haskell is lazy (non-strict) by default: nothing is evaluated until its result is actually demanded. This is the feature with no real C# equivalent — it enables infinite data structures and changes how you compose programs.

Lazy evaluation & infinite data
C#
// Eager by default; opt into laziness

// But arguments evaluate before the call:
int y = Const(1, Expensive()); // Expensive() runs
Haskell
-- Nothing evaluates until forced
ones :: [Int]
ones = 1 : ones            -- an infinite list!

first5 = take 5 ones       -- [1,1,1,1,1]
evens  = take 3 (filter even [0..])  -- [0,2,4]

-- Unused arguments are never evaluated:
const' x _ = x
y = const' 1 undefined    -- 1  (undefined never forced)
Lazy by defaultExpressions are thunks, computed only when something demands the result. That's why ones = 1 : ones and [0..] (all naturals) are legal: take 5 forces exactly five elements and no more. Laziness lets you separate "what data exists" from "how much of it we need."
const' 1 undefined == 1undefined is a value that crashes if forced — but const' ignores its second argument, so it never is. The flip side: thunks can pile up unevaluated (space leaks). seq and BangPatterns force strictness when you need it.

The Functor → Applicative → Monad ladder is Haskell's big abstraction, and you already use specialized versions of it in C#: LINQ's Select is fmap, and async/await is one monad's do-notation. Haskell just exposes the general interface.

fmap, bind, and do-notation
C#

    var a = await FetchA();
    var b = await FetchB(a);
    return a + b;

Haskell
-- fmap = map for ANY context
fmap (+1) (Just 3)     -- Just 4
fmap (+1) [1,2,3]     -- [2,3,4]

getTotal :: IO Int

  a <- fetchA          -- <- runs the action, binds result
  b <- fetchB a
  return (a + b)

-- desugars to: fetchA >>= \a -> fetchB a >>= \b -> return (a+b)
One ladder, many typesFunctor (fmap) maps inside a context; Applicative combines independent contexts; Monad (>>=, "bind") sequences dependent ones. Maybe, lists, Either, IO, and parsers are all monads — so the same do-block shape works across all of them.
do is just >>= sugara <- action binds the result of an effectful action; the block desugars to chained >>= (bind). C#'s await sequencing Task and LINQ's from … select are this exact idea hard-wired to one type; Haskell hands you the interface so you can write it for any monad.

Messages are gRPC's data types — the equivalent of your DTOs. The twist that trips up everyone coming from JSON: each field carries a number, and that number, not the name, is what goes on the wire.

gRPC has four method shapes, and there's only one knob to set them: the stream keyword, on the request side, the response side, both, or neither.

The four RPC types

        client.WatchOrders(req)

gRPC / proto
service OrderService {
  // 1. Unary — one request, one response
  rpc GetOrder (GetOrderRequest) returns (Order);

  // 2. Server streaming — one in, many out
  rpc WatchOrders (WatchRequest) returns (stream OrderEvent);

  // 3. Client streaming — many in, one out
  rpc BulkCreate (stream Order) returns (CreateSummary);

  // 4. Bidirectional — many ↔ many, interleaved
  rpc Chat (stream Message) returns (stream Message);

When to streamServer streaming for live feeds or large result sets you want incrementally; client streaming for uploads/aggregation; bidirectional for chat-like or interactive protocols. If you don't actually need it, unary is far easier to reason about and retry.

gRPC doesn't use HTTP status codes — it has its own status space. Every call completes with a status (a code, a message, and optional details), and failures come back as a status rather than a thrown-across-the-wire exception.

Two side-channel concerns every real service needs: passing headers (auth, trace IDs) and bounding how long a call may run. gRPC calls these metadata and deadlines, and the deadline is genuinely better than a client-side timeout.

Headers + timeout vs metadata + deadline
C# / REST
// REST: headers + a client-side timeout

    "authorization", token);

    TimeSpan.FromSeconds(2));

// timeout is local; downstream hops don't know
gRPC

// A deadline is an absolute time, and it PROPAGATES
var resp = await client.GetOrderAsync(
    req,
    headers: md,
    deadline: DateTime.UtcNow.AddSeconds(2));

// Server reads context.RequestHeaders and watches
// context.CancellationToken (deadline -> CANCELLED)
Deadlines beat timeoutsA deadline is an absolute "be done by" time, and gRPC propagates it to every downstream service in the call chain. When it passes, in-flight work is cancelled and the call fails DEADLINE_EXCEEDED — no orphaned work grinding on after the client already gave up.
Metadata = headersArbitrary key/value pairs that ride with a call — auth tokens, trace/correlation IDs, tenant info. Keys ending in -bin carry binary values; everything else is ASCII. It's the gRPC analog of HTTP headers, available on both the client call and the server context.

Because the field number is the wire identity, schema evolution has hard rules — but follow them and gRPC gives you backward and forward compatibility for free, which sloppy JSON APIs never guarantee.

Evolving the contract safely
JSON / REST
// JSON: additive changes are usually safe...
{
  "id": 1,
  "customer": "Acme",
  "note": "rush"      // adding a field: fine
}

// ...but nothing ENFORCES the contract.
// Renaming "customer" -> "buyer" silently
// breaks every consumer at runtime.
gRPC / proto
message Order {
  int64  id       = 1;
  string customer = 2;

  // ✅ ADD fields with NEW numbers — old clients
  //    just ignore what they don't know
  string note     = 5;

  // ❌ NEVER reuse/renumber a field number
  // ❌ NEVER change an existing field's type

  // Removed fields? Reserve number AND name so
  // they can't be recycled by accident:
  reserved 3, 4;
  reserved "discount";
}
The number is the contractCompatibility keys on field numbers, never names. Adding fields with fresh numbers is backward- and forward-compatible: old binaries ignore unknown fields and even pass them through untouched. Renaming a field is free (names aren't on the wire); changing its number or type is a breaking change.
reserved prevents footgunsWhen you delete a field, reserved its number and name so a later edit can't silently reuse them and misread old data. And note proto3 has no required fields by design — everything is optional at the wire level, which is precisely what makes safe evolution possible.

  sku: { name: 'Standard_LRS' }
  kind: 'StorageV2'
}
Terraform
# main.tf — declare the provider, then resources
terraform {
  required_providers {
    azurerm = { source = "hashicorp/azurerm", version = "~> 3.0" }
  }
}
provider "azurerm" { features {} }

resource "azurerm_storage_account" "sa" {
  name                     = "mystorage123"
  resource_group_name      = var.rg
  location                 = var.location
  account_tier             = "Standard"
  account_replication_type = "LRS"
}
Same idea, two dialectsBoth declare a storage account and let the engine figure out create-vs-update. Bicep is terser because it only targets Azure and inherits ARM's resource catalog directly. Terraform first declares a provider (here azurerm) — that pluggability is how the same language also provisions AWS, GCP, Cloudflare, Datadog, etc.
Declarative, not a scriptNeither file runs top-to-bottom like C#. You state the desired resources; the tool builds a dependency graph and converges to it. Re-running with no changes is a no-op — that's idempotence, the core IaC property.

A resource is the unit of both languages. The key difference is referencing: Bicep uses a symbolic name and reads properties off it directly; Terraform addresses resources by type.name.attribute and builds its dependency graph from those references.

Two dependent resources
Bicep
resource plan 'Microsoft.Web/serverfarms@2023-01-01' = {
  name: 'app-plan'
  location: location
  sku: { name: 'B1' }
}

resource site 'Microsoft.Web/sites@2023-01-01' = {
  name: 'my-api'
  location: location
  properties: {
    // reference creates an implicit dependsOn
    serverFarmId: plan.id
  }
}
Terraform
resource "azurerm_service_plan" "plan" {
  name                = "app-plan"
  resource_group_name = var.rg
  location            = var.location
  os_type             = "Linux"
  sku_name            = "B1"
}

resource "azurerm_linux_web_app" "site" {
  name                = "my-api"
  resource_group_name = var.rg
  location            = var.location
  # referencing plan.id wires the dependency

}
Dependencies are inferred from referencesYou almost never declare ordering by hand. plan.id (Bicep) and azurerm_service_plan.plan.id (Terraform) both tell the engine "build the plan first." Both have an explicit escape hatch — Bicep's dependsOn, Terraform's depends_on — but you rarely need it.

Parameterize so one template serves dev/staging/prod. Bicep splits param (external inputs) from var (computed locals); Terraform splits variable from locals — same distinction, different keywords.

Inputs & computed locals
Bicep
// External inputs (with defaults & validation)

@allowed(['Standard_LRS', 'Standard_GRS'])

// Computed local value
var saName = 'stg${env}${uniqueString(resourceGroup().id)}'

  name: saName
  sku: { name: skuName }

}
Terraform
# External inputs (with type, default, validation)
variable "env" {
  type    = string
  default = "dev"
}

  type    = string
  default = "Standard_LRS"

# Computed locals

}
param/var vs variable/localsSame split both sides: inputs come from outside (CLI, tfvars, pipeline) and can carry defaults + validation; locals are derived inside the template. String interpolation is '${...}' in Bicep and "${...}" in Terraform. Pass values in via --parameters (Bicep) or a .tfvars file (Terraform).

Modules package a chunk of infrastructure for reuse — a "network" module, a "web app + plan" module. Both languages treat a module as a file/folder you call with inputs and read outputs from.

Calling a reusable module

Reuse, with a registry on the Terraform sideBicep modules are local .bicep files (or pulled from an Azure Container Registry). This is the single biggest conceptual difference between the two — and the thing that bites newcomers. Terraform keeps a state file mapping config to real resources; Bicep keeps none, because Azure Resource Manager itself is the source of truth.

How each one knows what exists
Bicep — stateless
# Preview the diff against LIVE Azure
az deployment group create \
  --resource-group rg-app \
  --template-file main.bicep \
  --what-if            # shows +/-/~ changes

# Apply (incremental mode by default):
az deployment group create \
  --resource-group rg-app \
  --template-file main.bicep

# Truth = ARM. Bicep asks Azure what exists,
# every time. There is no local state file.
Terraform — stateful
# Compares config <-> state <-> real world
terraform plan      # shows the diff
terraform apply     # converges, updates state

# State lives in terraform.tfstate — keep it
# REMOTE & locked so a team doesn't corrupt it:
terraform {
  backend "azurerm" {
    resource_group_name  = "rg-tfstate"
    storage_account_name = "tfstate123"
    container_name       = "state"
    key                  = "app.tfstate"
  }
}
State is power and perilTerraform's state enables accurate diffs, resource renames (moved blocks), and clean destroys — but it can drift from reality, must be stored remotely + locked for teams, and contains secrets in plaintext. Bicep sidesteps all of this by re-reading ARM each run; the trade-off is no first-class destroy plan and weaker drift detection.
Incremental vs completeBicep/ARM deploys are incremental by default — resources not in the template are left alone (use --mode Complete to delete them). Terraform's model is "the config IS the world": anything you remove from .tf gets destroyed on the next apply. Different mental defaults for "what happens to things I stopped mentioning."

Need three subnets, or a resource only in prod? Both languages have loops and conditionals — declarative ones that expand into multiple resource instances rather than running imperatively.

Loops & conditionals
Bicep
param subnetNames array = ['web', 'app', 'db']
param deployFirewall bool = false

// Loop: one resource per array item
resource subnets 'Microsoft.Network/.../subnets@2023-05-01' = [
  for name in subnetNames: {
    name: name
    // ...
  }
]

// Conditional: deploy only if true
resource fw 'Microsoft.Network/azureFirewalls@2023-05-01' =
  if (deployFirewall) {
    name: 'app-fw'
    // ...
  }
Terraform
variable "subnet_names" {
  default = ["web", "app", "db"]
}
variable "deploy_firewall" { default = false }

# for_each: one instance per set member
resource "azurerm_subnet" "s" {
  for_each = toset(var.subnet_names)
  name     = each.value
  # ...
}

# Conditional via count = 1 or 0
resource "azurerm_firewall" "fw" {
  count = var.deploy_firewall ? 1 : 0
  name  = "app-fw"
}
Loops expand the graph, they don't "run"Bicep's [for x in xs: {...}] and Terraform's for_each/count both produce multiple resource instances at plan time. Prefer for_each (keyed by value) over count (keyed by index) in Terraform: removing the middle item of a count list renumbers everything after it and forces needless re-creates.
Conditionals are still declarativeBicep's = if (cond) {...} and Terraform's count = cond ? 1 : 0 decide whether a resource exists in the final graph — they're not runtime branches. There's no imperative "else run this other code"; you model presence/absence, not control flow.

You already ship containers; Kubernetes is how a cluster wants them described. You write declarative YAML manifests, kubectl apply them, and a control loop continuously reconciles the cluster toward that desired state — restarting, rescheduling, and scaling without you scripting any of it.

Every object has the same shape
The object skeleton
Kubernetes
# Declarative: this is what SHOULD exist
apiVersion: v1
kind: Pod
metadata:
  name: web
spec:
  containers:
    - name: web
      image: nginx:1.27
      ports:
        - containerPort: 80

# kubectl apply -f pod.yaml  → cluster converges
apiVersion / kind / metadata / specEvery manifest — Pod, Service, Ingress, anything — has the same four-part skeleton: which API (apiVersion), what object (kind), its identity (metadata), and the desired state (spec). You declare the what; the control loop owns the how and never stops reconciling.
apply, don't runkubectl apply -f is idempotent: it diffs your manifest against the live object and patches the difference. Re-applying unchanged YAML does nothing. This is the same desired-state-reconciliation mindset as the Terraform topic — the cluster is the running controller.

The Pod is the smallest deployable unit — one or more tightly-coupled containers that share a network namespace and storage. You'll rarely write a bare Pod in practice, and the next topic explains why.

A single container
Kubernetes
apiVersion: v1
kind: Pod
metadata:
  name: api
  labels:
    app: api
spec:
  containers:
    - name: api
      image: myregistry.azurecr.io/api:1.4.2
      ports:
        - containerPort: 8080
      env:
        - name: ASPNETCORE_ENVIRONMENT
          value: Production
You almost never apply a bare PodA lone Pod is mortal: if its node dies, nothing recreates it. Pods are meant to be managed by a higher-level controller (a Deployment) that owns the desired replica count and replaces failures. Think of the Pod as the thing being scheduled, not the thing you hand-write.

The Deployment is what you actually write for a stateless service. It owns a desired replica count, recreates Pods that die, and rolls out new image versions gradually with automatic rollback on failure — the self-healing that makes Kubernetes worth the YAML.

Keep N replicas running
Kubernetes
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api          # which Pods this owns
  template:             # the Pod spec to stamp out
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: myregistry.azurecr.io/api:1.4.2
          ports:
            - containerPort: 8080
selector + template is the core ideaThe Deployment's selector.matchLabels must match the Pod template's labels — that's the contract telling the Deployment "these Pods are mine." Change the image and kubectl apply; it performs a rolling update (new Pods up, old Pods drained) and rolls back automatically if the new ones fail their readiness checks.
Deployment → ReplicaSet → PodsA Deployment manages a ReplicaSet (which enforces the count), which manages the Pods. You interact with the Deployment; the layers beneath are how rollouts keep two versions side-by-side mid-deploy. kubectl rollout status/undo deployment/api drives it.

Pods are ephemeral and get new IPs constantly, so you never address them directly. A Service is a stable virtual IP + DNS name that load-balances across whichever Pods currently match its selector — service discovery, built in.

A stable address for a set of Pods
Kubernetes
apiVersion: v1
kind: Service
metadata:
  name: api
spec:
  type: ClusterIP      # internal-only (default)
  selector:
    app: api            # picks Pods by label
  ports:
    - port: 80          # the Service port
      targetPort: 8080  # the container port

# reachable in-cluster at  http://api  (DNS)
Selectors, not fixed IPsA Service has no idea which Pods exist — it just matches selector labels, so Pods can come and go and the Service keeps routing. Three Service types: ClusterIP (internal, default), NodePort (opens a port on every node), and LoadBalancer (provisions a cloud load balancer — on AKS, a real Azure LB with a public IP).
Built-in DNSEvery Service gets a cluster DNS name: api within the namespace, or api.shop.svc.cluster.local fully-qualified. That's how your services find each other — no service registry to run, no IPs to hardcode.

A LoadBalancer Service per app means a public IP per app. Ingress fixes that: one entry point that routes external HTTP by host and path to many internal Services — the reverse proxy you'd otherwise run as nginx, declared as YAML.

HTTP routing into the cluster
Kubernetes
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web
spec:
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api
                port: { number: 8080 }
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web
                port: { number: 80 }
Ingress is the rules; the controller does the workThe Ingress object is just routing rules — an Ingress controller (nginx-ingress, AGIC, Traefik…) running in the cluster actually implements them. One Ingress + one load balancer fronts many Services by host/path, and it's where you terminate TLS (referencing a Secret holding the cert).

Keep config out of your image. ConfigMaps hold non-secret settings, Secrets hold sensitive values, and both inject into Pods as environment variables or mounted files. The keys map cleanly onto .NET's __ config convention.

Externalized config & secrets
Kubernetes
apiVersion: v1
kind: ConfigMap
metadata:
  name: api-config
data:
  ASPNETCORE_ENVIRONMENT: Production
  Logging__LogLevel__Default: Information
---
apiVersion: v1
kind: Secret
metadata:
  name: api-secret
type: Opaque
stringData:
  ConnectionStrings__Db: "Server=sql;Password=..."

# In the Deployment's container spec:
#   envFrom:
#     - configMapRef: { name: api-config }
#     - secretRef:    { name: api-secret }
Secrets are base64, not encryptedA Secret's only difference from a ConfigMap is intent + base64 encoding — by default it is not encrypted at rest unless you enable encryption or back it with a real vault (on AKS, the Key Vault CSI driver). Use stringData so you write plaintext and Kubernetes encodes it; never commit a populated Secret to git.
The __ convention is a gift.NET configuration reads __ in env var names as section nesting, so Logging__LogLevel__Default binds straight to Logging:LogLevel:Default. ConfigMap/Secret keys drop into IConfiguration with zero glue code.

Two cross-cutting concepts hold a cluster together: Namespaces carve one physical cluster into isolated virtual ones (dev/staging/prod, or per-team), and labels + selectors are the loosely-coupled glue that lets objects find each other.

Isolation & the label glue
Kubernetes
apiVersion: v1
kind: Namespace
metadata:
  name: shop
---
# Labels are arbitrary key/values on every object
metadata:
  labels:
    app: web
    tier: frontend
    env: prod

# Selectors query them — this is how Services,
# Deployments, and you (kubectl) find Pods:
#   kubectl get pods -n shop -l app=web,env=prod
Labels are the universal join keyAlmost everything in Kubernetes is wired by label selectors, not hard references: a Service finds its Pods by label, a Deployment owns its Pods by label, network policies and kubectl filters all query labels. Namespaces add the isolation boundary — scoped names, RBAC, and resource quotas per namespace.

The last mile of production-readiness: tell Kubernetes how to know your container is healthy, and how much CPU/memory it needs. Probes drive self-healing and safe rollouts; resource requests/limits drive scheduling and autoscaling.

Health checks & resource bounds
Kubernetes
# inside the container spec:
livenessProbe:                  # restart if unhealthy
  httpGet: { path: /healthz, port: 8080 }
  initialDelaySeconds: 10
  periodSeconds: 10
readinessProbe:                 # gate traffic until ready
  httpGet: { path: /ready, port: 8080 }
resources:
  requests: { cpu: "100m", memory: "128Mi" }
  limits:   { cpu: "500m", memory: "256Mi" }
liveness ≠ readinesslivenessProbe failing → Kubernetes restarts the container (it's wedged). readinessProbe failing → it pulls the Pod out of the Service's rotation but leaves it running (warming up, or a dependency is down). Getting readiness right is what makes rolling updates zero-downtime — new Pods take traffic only once they say they're ready.
requests vs limits — and HPArequests is what the scheduler reserves to place the Pod; limits is the hard ceiling (exceed memory → OOM-killed; exceed CPU → throttled). Requests also feed the HorizontalPodAutoscaler: kubectl autoscale deployment api --cpu-percent=70 --min=3 --max=10 scales replicas against the requested CPU.

Docker packages your app plus everything it needs into an image — an immutable, layered snapshot — and runs instances of it as containers. One build runs identically on your laptop, CI, and the cluster. This is the layer everything else here (Compose, Kubernetes) sits on top of.

The core lifecycle
Docker CLI
# Build an image from a Dockerfile in .
docker build -t myapi:1.4.2 .

# Run a container from that image
docker run -d -p 8080:8080 --name api myapi:1.4.2

# Observe & manage running containers
docker ps                 # what's running
docker logs -f api        # follow stdout/stderr
docker exec -it api sh    # shell inside it
docker stop api && docker rm api
Image vs containerAn image is the read-only template (like a class); a container is a running instance of it (like an object). One image → many containers. Images are immutable — you don't edit them, you build a new tag. Containers are disposable — treat them as cattle, not pets.
The same artifact, everywhereThe whole point: the image you build locally is byte-for-byte what runs in CI and prod. "Works on my machine" stops being a sentence, because your machine and prod run the identical image.

A Dockerfile is the recipe for an image — an ordered list of instructions, each producing a cached layer. For .NET the idiomatic pattern is a multi-stage build: compile in the heavy SDK image, then copy only the published output into a slim runtime image.

A .NET multi-stage Dockerfile
Dockerfile
# ---- build stage: the big SDK image ----
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
# Copy csproj first & restore — caches deps
COPY *.csproj ./
RUN dotnet restore
# Then copy the rest & publish
COPY . .
RUN dotnet publish -c Release -o /app

# ---- runtime stage: the slim aspnet image ----
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY --from=build /app .
EXPOSE 8080
ENTRYPOINT ["dotnet", "MyApi.dll"]
Order instructions by how often they changeEach instruction is a cached layer; a layer rebuilds only if it or something above it changed. Copying *.csproj and restoring before copying source means a code change doesn't re-download NuGet packages — the restore layer stays cached. Put the volatile stuff (your source) last.
Multi-stage = small, safe imagesThe final image is built FROM the runtime image and only COPY --from=build brings over the published output — the SDK, your source, and build tools never ship. Smaller attack surface, faster pulls. Add a .dockerignore (bin/, obj/, .git/) so the build context stays lean.

Images are named repository:tag and live in registries. On your stack that's Azure Container Registry; push there and AKS pulls from it. Under the hood an image is a stack of content-addressed layers shared across images.

Tag, push, pull
Docker CLI
# Tag a local image for a registry (ACR)
docker tag myapi:1.4.2 myregistry.azurecr.io/myapi:1.4.2

# Authenticate, then push / pull
az acr login -n myregistry
docker push myregistry.azurecr.io/myapi:1.4.2
docker pull myregistry.azurecr.io/myapi:1.4.2

# Inspect the layers (and their sizes)
docker history myapi:1.4.2
Layers are shared & cachedEach image is a stack of layers identified by content hash. Pull a second image that shares a base layer and Docker reuses what it already has — only the new layers transfer. That's why a tiny code change ships a tiny push: just the top layers differ.
Pin real versions, never trust latestTags are mutable pointers — latest can mean a different image tomorrow, which wrecks reproducibility. Tag with an explicit version (or the git SHA) so a deploy is pinned to an exact, immutable artifact. For true immutability, reference the image by its @sha256: digest.

Everything about how a container runs — ports, config, storage, restart behavior — is flags on docker run, not baked into the image. Same image, different runtime wiring per environment.

Running & interacting with containers
Docker CLI
docker run \
  -d \                              # detached (background)
  --name api \
  -p 8080:8080 \                    # publish host:container
  -e ASPNETCORE_ENVIRONMENT=Production \  # env var
  -v appdata:/app/data \            # mount a volume
  --restart unless-stopped \        # restart policy
  myapi:1.4.2

# Peek inside a running container
docker exec -it api sh
docker logs -f api
Image = what, run flags = howThe image is the immutable app; -p, -e, -v, and --restart are environment wiring supplied at launch. This split is exactly what Kubernetes formalizes — the container image is the same, and the Pod spec carries the ports/env/volumes/restart policy as declarative YAML.

Real apps are several containers — API, database, cache. Compose declares the whole set in one YAML file and brings them up together with one command. It's the local-dev workhorse, and a clear stepping stone to Kubernetes manifests.

A multi-service stack
docker-compose.yml
services:
  api:
    build: .                 # build from local Dockerfile
    ports:
      - "8080:8080"
    environment:
      ASPNETCORE_ENVIRONMENT: Production
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - dbdata:/var/lib/postgresql/data

volumes:
  dbdata:                    # named volume, persists

# docker compose up -d   ·   docker compose down
One file, one network, names as DNSCompose puts every service on a shared network and gives each a DNS name, so the API reaches Postgres at db:5432 — no IPs, no links. depends_on controls start order. It's the same declarative, multi-service idea Kubernetes scales to a cluster.
Compose is dev; K8s is prodCompose is unbeatable for spinning up a dev environment, but it runs on one host and has no real self-healing, rolling updates, or autoscaling. When you outgrow a single machine, the same containers move to Kubernetes — Compose is the on-ramp, not the destination.

By default a container is isolated. Put containers on a user-defined network and they get automatic DNS by name; publish a port only when something outside the host needs in. This name-based discovery is exactly what Compose and Kubernetes Services automate.

Networks, DNS & publishing
Docker CLI
# A user-defined network gives DNS-by-name
docker network create app-net

docker run -d --name db  --network app-net postgres:16
docker run -d --name api --network app-net myapi:1.4.2

# api reaches the database at hostname "db".
# No -p needed for container-to-container traffic.

# Publish a port ONLY for external access:
docker run -d -p 8080:8080 --network app-net myapi:1.4.2
Internal DNS vs published portsContainers on the same user-defined network talk to each other by name over a private bridge — that's service discovery for free. -p host:container is a separate concern: it pokes a hole from the host into the container, needed only for traffic coming from outside. Default bridge network is the exception — it has no automatic DNS, so always create your own.

Containers are ephemeral — delete one and its writable layer is gone. Anything that must outlive a container (a database's files) or be shared with the host (your source during dev) lives outside it, in a volume or a bind mount.

Persisting & sharing data
Docker CLI
# Named volume — Docker-managed, persists
docker volume create dbdata
docker run -v dbdata:/var/lib/postgresql/data postgres:16

# Bind mount — a host path, great for dev
docker run -v "$(pwd)/src":/app/src myapi:1.4.2

# tmpfs — in-memory, vanishes on stop
docker run --tmpfs /tmp myapi:1.4.2
Volume vs bind mountA named volume is managed by Docker and is the right call for real data (databases) — portable and backup-friendly. A bind mount maps a specific host directory in, ideal for mounting your source for live-reload in dev. Rule of thumb: volumes for data you keep, bind mounts for files you're editing.
The writable layer is disposableWrites that don't go to a volume land in the container's thin writable layer and die with the container. Never store anything you care about there. In Kubernetes this same lesson returns as PersistentVolume/PersistentVolumeClaim — Pods are even more disposable than containers.
